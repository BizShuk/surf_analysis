# 影片格式決策 (Video Format Decision)

在進行衝浪影片的生物力學分析時，理解影片的封裝格式與編碼格式對於確保分析管線的穩定性至關重要。

## 背景觀念 (Background Concepts)

理解影片格式最生活化的比喻如下：

- `封裝格式 (Container)` 就像是快遞的外部紙箱（例如 `.mp4`、`.webm`）。
- `編碼格式 (Codec)` 則是把內容物壓緊放進箱子裡的壓縮技術（例如 `H.264`、`AV1`）。

真正決定影片檔案大小與畫質的，主要是編碼技術。
以下拆解目前最普及的格式，以及技術上較有效率的選擇：

### 1. 普及格式：相容性高的安全選擇

不論是 `YouTube`、`Instagram`、`TikTok` 還是 `X (Twitter)`，目前相容性最高的組合為：

- `格式組合`：`MP4` 封裝格式搭配 `H.264 / AVC` 編碼。
- `100% 相容性`：不論是舊型顯示設備、各種瀏覽器，還是最新款的手機，
  都能完美播放，完全不會有無法開啟的風險。
- `上傳首選`：它是各大社群平台官方推薦的 `輸入格式 (Ingest Format)`。
  雖然壓縮效率在今天看來不算最高，但因為最安全，
  平台後台在進行 `轉碼 (Transcoding)` 時最不容易出錯。

### 2. 高效格式：高畫質與省流量的選擇

若追求以最小的檔案體積裝下最高的畫質（如 4K 或 8K），目前常見的高效編碼有以下兩大陣營：

- `AV1 (AOMedia Video 1)`：現代網路串流省流量之王。
    - `效率分析`：檔案體積比傳統 `H.264` 大幅減少約 50% 至 65%，
      也比 `HEVC` 還要再節省 30% 的頻寬。
    - `應用場景`：`YouTube 4K/8K` 影片與 `Netflix` 的核心串流格式。
      它是免授權費的開放標準，各大科技巨頭都在普及它。
      不過，只有較新的硬體晶片才能流暢進行 `硬體解碼 (Hardware Decoding)`。
- `HEVC / H.265 (High Efficiency Video Coding)`：行動錄影與蘋果生態系主力。
    - `效率分析`：壓縮率大約是 `H.264` 的 2 倍（相同畫質下檔案大小直接砍半）。
    - `應用場景`：這是 iPhone 錄製 4K 影片、高畫質動態照片的預設格式，
      也是 4K 藍光光碟的標準。缺點是它需要高昂的專利授權費，
      因此在部分 Windows 電腦或開源瀏覽器上可能需要額外安裝解碼包。

### 3. 實用總結

- `上傳社群或分享給他人`：請選擇 `MP4 (H.264)`，保證相容性，不需冒無法播放的風險。
- `自行保存 4K 影片或進行備份`：選擇 `HEVC (H.265)`，能節省大約一半的硬碟空間。
- `技術前瞻`：雖然新一代的 `AV2` 編碼已經發表，號稱比 `AV1` 還能再省 30% 流量，
  但要等到硬體設備全面支援，還需要數年的時間。

## 串流格式 vs. 本機分析需求 (Streaming Formats vs. Local Analysis Requirements)

上述一般性的影片格式說明偏向「串流平台播放與上傳」場景。
然而，`surf_analysis` 是一個 `離線批次 (Offline Batch)` 的 CLI 工具，
兩者的技術考量重點不同。

`surf_analysis` 對影片編碼與封裝格式的關注點是「`OpenCV` 與 `MediaPipe` 能不能順利解碼」，
而不是「觀眾瀏覽器能不能播放」或「上傳到社群省不省流量」。

下表對比了串流場景與本專案實際需求的差異：

| 場景                          | 原文聚焦                 | 本專案實際需求                   |
| :---------------------------- | :----------------------- | :------------------------------- |
| `YouTube` / `Netflix` 4K 串流 | 平台轉碼、頻寬成本       | 完全無關                         |
| iPhone / Android 硬解         | 終端播放相容性           | 與本機解碼無關                   |
| 社群上傳最佳化                | `AV1` 與 `HEVC` 編碼選擇 | 跟輸入端無關                     |
| 容器與編碼的比喻              | 觀念建立                 | 唯一直接相關的觀念               |
| `MP4` 搭配 `H.264` 最相容     | 跨平台播放               | 可延伸為「解碼最安全穩定」的組合 |

這份內容對專案真正有價值的部分，只有「封裝與編碼的觀念比喻」，
以及「`MP4` 搭配 `H.264` 是事實標準」這兩個結論：

- 輸入端影片來自 iPhone（`MOV` 封裝格式與 `HEVC` 編碼）是已知的痛點
  （例如虛擬環境下解碼器不相容的問題）。
- 分析管線需要 `OpenCV` 讀取影片，而 `H.264` 在 `MP4` 容器中是 `FFmpeg`
  與 `OpenCV` 預設解碼最穩定的組合。
- 本專案已進行過 `MOV` 轉 `MP4` `串流複製 (Stream Copy)` 的實作驗證。

其餘關於 `AV1`、`Netflix`、網頁播放器、硬體解碼與 4K 串流的內容，
對於這個 CLI 工具並無實質幫助。

## 影片格式關注樹 (Video Format Decision Tree)

以下為本專案設計的影片格式指南結構：

- `輸入端 (Input)`
    - 支援封裝格式：`MP4`、`MOV`、`AVI`（`FFmpeg` 預設解碼鏈）。
    - 支援編碼格式：`H.264`（最穩定）、`HEVC`（iPhone 預設，需在環境中驗證相容性）。
    - 不支援或需轉檔：`ProRes`、`VP9`、`AV1`（`MediaPipe` 較新版本才逐步支援）。
- `處理端 (Pipeline)`
    - `MediaPipe Pose`：內部解碼成 RGB 圖像影格 (Frame)，不依賴封裝格式。
    - `OpenCV` `cv2.VideoCapture`：使用 `FFmpeg` 後端 (Backend)，編碼器效能為主要瓶頸。
    - 效能瓶頸：解碼速度大於編碼效率（因為影片只會進行一次性讀取分析）。
- `輸出端 (Output)`
    - 標註影片：`MP4` 封裝格式搭配 `H.264` 編碼（`cv2.VideoWriter` 預設 `fourcc='mp4v'`）。
    - 命名規範：`<file_name>.annotated.<ext>`（依據 `CLAUDE.md` 規定，其中 `<ext>` 代表副檔名）。

## 核心洞察 (Core Insights)

- `工程世界的差異`：離線分析管線與串流平台是兩種完全不同的工程領域。
  前者關心「單機一次性解碼的吞吐量 (Throughput) 與相容性」，
  後者關心「CDN 頻寬成本、終端裝置硬體解碼能力與社群播放相容性」。
  套用串流思維到 CLI 工具會使技術焦點失焦。
- `環境相容性隱憂`：`surf_analysis` 依賴 `FFmpeg` 作為解碼後端，
  代表 any `FFmpeg` 不支援的編碼器都可能導致 `cv2.VideoCapture` 階段直接失敗。
  這也是為什麼 `MOV` 搭配 `HEVC` 的影片在某些 Linux 環境下，
  需要額外安裝解碼授權或函式庫。
- `文件策略`：專案內的技術說明應緊扣「這個工具的失敗模式」而非「產業趨勢」，
  否則會增加維護負擔卻無法解決實際的解碼錯誤問題。
